# Agent Instructions

## Communication style
No social etiquette is needed, we are here to do work. Be polite but neutral. Develop the work products without commentary. 

## Scope
Do not do anything that was not explicitly requested. Do not speculate about scope — if something was not asked for, do not add it.

Do not suggest next steps, do not prompt me to do things. But do mount constructive criticism of the current state of the work products.

## Goal
To informally specify a ReSTful "back-end-for-front-end" web service for a proposed system.

The implementation will follow these principles:
* Command-Query Separation
* DRY
* Hexagonal architecture
* Strong ReSTful design: prefer meaningful URIs

## Strategy
From an informal, natural language specification of required behavious you wil derive a specification using Alloy 6. From the examples generated by this model you will generate a suite of tests. You will build a bare-bones implementation of the service and demonstrate that the tests pass.  

If any step is difficult try up to three different approaches then stop and seek guidance.

We will work in a style that is:
* iterative, we will expect to work again on what we have worked on before; nothing needs to be perfect first time
* incremental, we will perform many rounds of work, with the scope of the spec and the capability of the system increasing over time
* evolutionary, we hope and expect to change the spec, tests, and systems over time to be more fit, especially to be easier to change
* double-loop, we will not only improve the work products, we will also improve how we work

## Tools 
* version control: git
* formal specification: Alloy 6
* testing: Playwright
* implementation: TypeScript, Node, React, Vite

## Workflow
1. after the user makes changes to `specification/requirements.md`:
   1. update the model in files under `specification/alloy` to reflect the natural-language spec as closely as possible
   2. iterate the model until Alloy reports both that every `check` reports `UNSAT` (meaning "no counterexamples found") and every `run` reports `SAT` (examples found)
   3. update the test suite to reflect any changes to the library of examples
   4. update the implementation until all tests pass
2. After a round, inform me of opportunities to improve:
   1. the implementation
   2. the tests based on the current state of the implementation
   2. the formal spec based on the current state of the tests
   3. the natural-language spec based on the current state of the formal spec
